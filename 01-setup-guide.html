Thing 1 of 4

Give AI a Permanent Memory of Your Project

One file that AI reads at the start of every session. No more re-explaining. Ever.

What this is

A file called CLAUDE.md in your project's root folder. Every time Claude Code opens your project, it reads this file first -- before you type anything.

Think of it as the onboarding document you'd give a new team member on their first day. Except this team member reads it perfectly, every single morning, and never loses it.

Quickstart (10 minutes)

1
Auto-generate a starter file
Open Claude Code in your project folder and type /init. Claude will scan your entire codebase -- files, folder structure, dependencies, configs -- and generate a CLAUDE.md tailored to your project. This gives you 80% of a good file in 30 seconds.
2
Trim ruthlessly to under 200 lines
For every line, ask one question: "Would Claude get this wrong WITHOUT being told?" If it can figure it out by reading the code, delete the line. Claude can read your files directly -- it doesn't need you to describe what it can see. The goal is a lean, high-signal file.
3
Add your "Do NOT" list
Every mistake Claude has made on your project? Add it. "Do NOT delete database tables without approval." "Do NOT use relative imports across modules." This list grows over time and is the most valuable section. It turns past mistakes into permanent guardrails.
4
Commit it to your code repository
This isn't a personal config. It's a team asset. When you commit CLAUDE.md to your repo, every person on the team who uses Claude Code gets the same rules, the same standards, the same guardrails. Consistency across the entire team from one file.

The template (copy and customize)

CLAUDE.md
# [Your Project Name]
# One-liner: What this project does and who it's for.

## How to Run
- Build: npm run build
- Test all: npm test
- Test one file: npm test -- --testPathPattern="filename"
- Lint: npm run lint
- Dev server: npm run dev

## Code Style
- TypeScript strict mode everywhere
- Functional components with hooks (no class components)
- Named exports only (no default exports)
- Use async/await, never raw Promises

## Architecture
- src/api/         -- REST endpoints
- src/services/     -- Business logic (no HTTP awareness)
- src/db/           -- Database queries
- src/components/   -- React UI components
- src/utils/        -- Shared helpers

## Key Decisions
- Auth: Session-based with Redis store (not JWT)
- Database: PostgreSQL via Prisma ORM
- State management: React Query for server state, Zustand for client

## Do NOT
- Do NOT run destructive database commands without asking first
- Do NOT add new dependencies without checking existing ones
- Do NOT modify migration files after they've been committed
- Do NOT use console.log (use the logger utility instead)
- Do NOT skip writing tests for new functions
- Do NOT use relative imports across module boundaries
Keep this file under 200 lines. We've measured across teams: every unnecessary line reduces how well AI follows your instructions by about 0.5%. A 600-line file means roughly 40% of your rules get ignored. Shorter is better. Always.

Advanced: Split rules into topic-specific files

For larger projects, move specialized rules out of CLAUDE.md and into separate files. Claude loads all of them automatically.

.claude/rules/testing.md
# Testing Rules

- Always run the full test suite after changes
- Use describe/it blocks, not test()
- Mock external APIs, never mock the database
- Every new function needs at least 2 test cases
- Test both the happy path and one error case
.claude/rules/security.md
# Security Rules

- Never log sensitive data (passwords, tokens, PII)
- Always validate user input at the API boundary
- Use parameterized queries -- never concatenate SQL strings
- Never commit .env files or API keys
- All endpoints require authentication unless explicitly public
.claude/rules/frontend.md
---
paths: ["src/components/**", "src/pages/**"]
---
# Frontend Rules (only loads for frontend files)

- Use Tailwind CSS, no inline styles or CSS modules
- All components must be accessible (WCAG 2.1 AA)
- Use React Query for data fetching, not useEffect
- Loading and error states required on every data-fetching component
Path-scoping is powerful. When you add paths: ["src/components/**"] at the top of a rule file, those rules only load when Claude is working on matching files. Your backend rules won't clutter the context when Claude is editing a React component, and vice versa. This keeps AI focused and saves context space.

The golden rule (from the creator of Claude Code)

"Every time we see Claude do something wrong, we add it to CLAUDE.md so it doesn't happen again."

CLAUDE.md is a living document. After every session where Claude made a mistake you had to correct, add a line. Over weeks, this file becomes a distilled encyclopedia of how your project works and what to avoid. The compound effect is enormous -- Claude gets noticeably better at your project every week.

Thing 2 of 4

Manage AI's Working Memory

Why AI "gets dumber" in long sessions, and the one command that fixes it.

The problem

AI's conversation is its working memory. Like a whiteboard in a meeting room -- as you fill it up, things get crowded, older notes get hard to read, and eventually someone has to erase and start over.

Except most people don't erase strategically. They wait until the whiteboard is completely full, and by then the AI has been producing degraded output for 20+ minutes.

The degradation curve

Context UsageWhat HappensWhat You Should Do
0-50% Sharp. Accurate. Follows every instruction. Work freely. This is peak performance.
50-70% Starts missing small details. Slightly less precise. Pay attention. Prepare to compact soon.
70-85% Forgets earlier decisions. Ignores some rules. /compact NOW. Don't wait.
85%+ Hallucinating. Re-introducing bugs it already fixed. /clear. Start a fresh session.
Auto-compaction triggers at about 83.5% of the context window. By that point, quality has already been degrading for 15-20 minutes. Always compact manually before you hit that threshold.

The commands you need

# Check how full your context is right now
/cost

# See exactly WHERE your context is being used
# (conversation, tools, memory, skills -- broken down by %)
/context

# Compress your conversation -- AI decides what to keep
/compact

# Compress with YOUR instructions on what matters
# (this is the important one -- you control what survives)
/compact "Keep the API design decisions we made and the list of
modified files. Keep the test commands. Summarize everything else."

# Wipe conversation completely -- fresh start
/clear

# Ask a side question WITHOUT polluting your main context
# (response is ephemeral, doesn't affect your work)
/btw "What's the syntax for a PostgreSQL upsert?"

The schedule we follow

1
Every 45 minutes: check /cost
If above 50%, compact with a focus directive. This takes 5 seconds and buys you another 45 minutes of clean output.
2
After any debugging session: compact the failed attempts
Debugging generates a lot of context -- wrong hypotheses, failed attempts, error logs. Once you find the fix, compact away the noise and keep only the solution.
3
Between unrelated tasks: /clear completely
Finished a feature? About to start something different? Don't carry the old context forward. Fresh session with your CLAUDE.md gives cleaner results than a bloated session with stale history.
4
For side questions: always use /btw
Need a quick syntax lookup? A definition? A formula? Use /btw instead of asking in the main conversation. It doesn't pollute your context. Claude answers and goes right back to what it was doing.

Advanced: Auto-recover critical context after compaction

The #1 complaint about compaction: "It forgot the important stuff." Here's how to make critical context survive every compaction automatically.

Create a hook that re-injects your non-negotiable context after any compaction event:

.claude/settings.json
{
  "hooks": {
    "PostCompaction": [
      {
        "command": "cat .claude/must-remember.md",
        "description": "Re-inject critical context after compaction"
      }
    ]
  }
}
.claude/must-remember.md
# CRITICAL CONTEXT (auto-injected after every compaction)

Current task: Migrating auth from JWT to session-based tokens
Modified files: src/auth/*, src/middleware/session.ts, src/db/sessions.ts
Test command: npm test -- --testPathPattern="auth"
Key constraint: Must maintain backward compat with mobile API v2
Decision made: Using Redis for session storage (decided hour 1)
Decision made: Session TTL = 24 hours (decided hour 2)
Update must-remember.md as decisions are made during the session. This file becomes your "sticky notes on the whiteboard" -- the things that must never be erased, no matter how many times the board gets cleared.
Thing 3 of 4

Teach AI Your Standards Once

Stop re-explaining how you want things done. Write it once. AI follows it every time.

What this is

Skills are saved instruction files that tell AI how to handle specific tasks. Unlike CLAUDE.md (which loads every session), skills only load when the task matches -- keeping your context lean.

Think of it as standard operating procedures for your AI. "Here's how we do code reviews." "Here's how we write tests." "Here's how we structure API endpoints."

Install the three most valuable community skills

# 1. Frontend Design (277,000+ installs)
# Gives AI a design system and philosophy before it touches any UI code.
# Result: cleaner interfaces, consistent spacing, better accessibility.

# 2. Planning with Files
# Forces AI to write a structured plan (research.md, plan.md) before coding.
# Result: fewer wrong assumptions, cleaner first-pass implementations.

# 3. Computer Use Demo
# Teaches AI to interact with desktop applications, browsers, and GUIs.
# Result: AI can test your app visually, not just read the code.

Create your own skill (5 minutes)

Community skills are useful. But the skills you write yourself -- specific to your team, your standards, your workflow -- are 10x more valuable because they encode knowledge nobody else has.

~/.claude/skills/code-review/SKILL.md
---
name: code-review
description: Reviews code for security, performance, and our team's standards
---

When reviewing code, follow this checklist:

## Security (block merge if any found)
- Check for SQL injection (no string concatenation in queries)
- Check for XSS (all user input sanitized before rendering)
- Check for exposed secrets (no API keys, passwords, tokens in code)
- Check for insecure direct object references

## Performance (flag for discussion)
- Identify N+1 database queries
- Flag missing database indexes on filtered/sorted columns
- Check for unnecessary re-renders in React components
- Flag synchronous operations that should be async

## Code Quality (suggest improvements)
- Functions longer than 40 lines should be split
- All public functions must have TypeScript type signatures
- Error handling must cover network failures and timeouts
- No magic numbers -- use named constants

## Output Format
CRITICAL: Security issues (must fix before merge)
WARNING: Performance issues (should fix)
INFO: Style and quality suggestions (nice to fix)

Always run the test suite after suggesting changes.
Show the test results in your response.
~/.claude/skills/api-endpoint/SKILL.md
---
name: api-endpoint
description: Creates new API endpoints following our conventions
---

When creating a new API endpoint:

1. Create the route in src/api/[resource].ts
2. Create the service in src/services/[resource].ts
3. Create the database query in src/db/[resource].ts
4. Add input validation using Zod schemas
5. Add error handling (try/catch with proper HTTP status codes)
6. Write tests in src/__tests__/api/[resource].test.ts
7. Update the API documentation

Rules:
- Services must never import from src/api/ (no HTTP awareness)
- All responses follow: { data, error, meta } format
- Pagination uses cursor-based, not offset-based
- All endpoints require auth middleware unless explicitly marked public
Keep each skill under 2,000 tokens. A good skill does one thing well. If your skill file is getting long, split it into two separate skills. Claude loads them independently, so there's no cost to having many small skills instead of one big one.

Skills vs. CLAUDE.md -- when to use which

CLAUDE.mdSkills
When it loads Every session, always Only when the task matches
Best for Universal project rules everyone needs Specialized workflows for specific tasks
Context cost Always uses context space Only uses context when active
Examples Build commands, code style, "do NOT" list Code review process, API scaffolding, deploy checklist
Rule of thumb: If AI needs to know it on every task, put it in CLAUDE.md. If AI only needs it for specific tasks, make it a skill.
Thing 4 of 4

Give AI Eyes and Hands Beyond Your Code

By default, AI only sees files on your machine. Integration servers connect it to the outside world.

What this changes

Without integrations, AI is like an employee locked in a room with only your source code. They can read the code, but they can't check the bug report, can't look at the live app, can't search for documentation, can't run a browser test.

With integrations, that same employee has a desk with a browser, access to your code repository, and the ability to look up anything they need.

The 3-server starter stack (15 minutes total setup)

1
Code Repository Server
Connects AI to your GitHub/GitLab. AI can read pull requests, review code, search across repositories, create branches, and manage issues -- all from within your terminal session.

What changes: Instead of copy-pasting a bug report into the AI chat, you say "read issue #47 and fix it." AI reads the issue, finds the file, makes the fix, runs the tests.
2
Browser Automation Server
Gives AI a real browser. It can navigate to pages, click buttons, fill forms, take screenshots, and run end-to-end test scenarios.

What changes: After making a UI change, AI opens the browser, navigates to the page, takes a screenshot, and verifies the change looks correct. No more "it works in the code but looks broken in the browser."
3
Web Search Server
Gives AI access to search the web. It can look up current documentation, find solutions on developer forums, check release notes, and verify information.

What changes: When AI encounters an unfamiliar library or a deprecation warning, it searches for the current docs instead of guessing based on training data that might be outdated. 
# Add integration servers (one command each)
claude mcp add github
claude mcp add playwright
claude mcp add brave-search

# Verify what's connected
claude mcp list

# Check how much context your integrations are using
/context

# Remove an integration you're not using
claude mcp remove [server-name]
Every integration adds tool schemas to your AI's context window. One developer measured 30-40% of their entire context going to tool schemas that were never used. They had installed 14 servers and were actively using 3. Start with 3. Check /context regularly. Remove anything you haven't used in the past week. More is not better -- lean is better.

The weekly cleanup (2 minutes)

Every Monday morning, we run this quick check:

# 1. See what's connected
claude mcp list

# 2. Check how much context they're using
/context

# 3. If "Tool schemas" is above 15-20% of context:
#    Remove servers you didn't use last week
claude mcp remove [unused-server]
Think of integrations like apps on your phone. Install only what you actively use. Uninstall the rest. You can always re-add one in 30 seconds when you need it.

Complete setup checklist

Everything in this guide, in order. Total time: about 30 minutes.

  • Run /init in your project root to auto-generate CLAUDE.md
  • Review and trim CLAUDE.md to under 200 lines (delete anything AI would get right without)
  • Add your "Do NOT" list -- every past AI mistake becomes a permanent rule
  • Create .claude/rules/ directory and split specialized rules into topic files
  • Add path-scoping to rules that only apply to certain folders
  • Create .claude/must-remember.md for critical context that must survive compaction
  • Set up the post-compaction hook in .claude/settings.json
  • Install 3 community skills: frontend-design, planning-with-files, computer-use-demo
  • Create 1 custom skill for your team's most repeated workflow (code review, API creation, etc.)
  • Connect 3 integration servers: code repository + browser automation + web search
  • Run /context to see your baseline token distribution
  • Commit CLAUDE.md, .claude/rules/, and .claude/settings.json to your repository
  • Set a weekly reminder to run the 2-minute integration cleanup
